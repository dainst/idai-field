## 한 문단 요약

이 논문은 iDAI.field가 왜 만들어졌고 어떤 기술적 선택을 통해 현재의 Field Desktop, Field Hub, iDAI.field Web 구조로 발전했는지를 설명한다. 핵심 문제의식은 고고학 현장 기록이 조사 유형, 국가별 행정 지침, 프로젝트별 기록 관행에 따라 크게 달라지지만, 동시에 장기 보존과 비교 연구를 위해서는 공통 데이터 구조가 필요하다는 데 있다. iDAI.field는 고정된 핵심 데이터 모델과 프로젝트별 확장을 함께 지원하고, 오프라인 현장 작업과 사후 동기화를 견디는 구조를 목표로 TypeScript, Electron, PouchDB/CouchDB 같은 오픈소스 기술을 선택했다.

## 섹션별 해설

### 1. 문제의식

DAI는 발굴, 지표조사, 건축 조사처럼 서로 다른 유형의 현장 연구를 수행한다. 각 연구는 기록 항목과 흐름이 다르고, 국제 공동 조사에서는 현지 문화유산 기관의 지침도 따라야 한다. 논문은 이런 조건에서 현장 기록 소프트웨어가 충족해야 할 요구를 다섯 가지로 정리한다.

- 충분히 유연하고 조정 가능한 데이터 모델
- 지역, 시대, 주제를 넘어 비교 가능한 표준 핵심 스키마
- 다국어 데이터 입력
- 온라인과 오프라인을 모두 지원하고 지연 동기화에 강한 구조
- 연구 성과와 1차 자료를 함께 공개하기 쉬운 출판 흐름

이 지점은 한국 고고학용 야장을 설계할 때도 중요하다. 현장별 양식은 달라도, 최소 공통 항목은 비교 가능하게 유지해야 한다.

### 2. 개발사

iDAI.field의 초기 버전은 2006년부터 FileMaker 기반으로 개발되었다. 당시에는 Pergamon 발굴 프로젝트 요구에 맞춰 시작했고, 이후 여러 프로젝트가 복제해 사용하는 방식으로 확장되었다. 하지만 프로젝트별 수정 사항을 다시 기본 복제본에 반영하는 과정이 복잡했고, 폐쇄적 파일 형식과 라이선스 비용, 버전 업그레이드에 따른 마이그레이션 문제가 컸다.

2009년 이후에는 웹 기술 기반 후속 시스템을 구상했지만, 매우 넓은 범위를 한 번에 포괄하려다 실용 애플리케이션으로 이어지지 못했다. 2016년 DAI는 새 프로젝트로 다시 시작하면서 범위를 현실화했고, 현장 기록에 필요한 핵심 조건을 중심으로 iDAI.field를 재설계했다.

### 3. 기술 선택

현재 iDAI.field는 오픈소스 웹 기술을 바탕으로 한다. Field Desktop은 TypeScript, Angular, Leaflet, Electron을 사용해 Windows, macOS, Linux에서 같은 기능과 레이아웃을 제공한다. Electron을 쓰기 때문에 사용자는 별도 서버나 데이터베이스를 직접 구성하지 않고 설치형 데스크톱 앱처럼 사용할 수 있다.

오프라인 작업과 사후 동기화를 위해 PouchDB를 사용하고, 중앙 동기화에는 CouchDB 계열 복제 기능을 활용한다. NoSQL 문서 구조를 택한 이유는 프로젝트마다 달라지는 데이터 모델을 JSON 문서로 유연하게 표현하기 쉽기 때문이다.

### 4. 데이터 모델

논문에서 가장 중요한 설계 포인트는 데이터 모델을 `core`, `library`, `project-specific` 세 층으로 나누는 방식이다.

- `core`: 모든 프로젝트가 공유하는 최소 핵심 구조. 예를 들어 식별자, 지오메트리, 주요 상위 범주와 관계가 여기에 속한다.
- `library`: 반복적으로 쓰이는 하위 범주와 필드를 모아둔 재사용 가능한 구성. 예를 들어 유물의 하위 범주나 자주 쓰는 속성 필드를 제공한다.
- `project-specific`: 개별 프로젝트의 특수한 필요를 반영하는 사용자 정의 범주와 필드.

이 구조는 표준화와 현장 적응성 사이의 균형을 잡기 위한 장치다. 한국형 야장에서도 국가기관, 발굴조사기관, 대학 조사단, 박물관 프로젝트가 공통으로 쓰는 핵심 항목과 현장별 확장 항목을 분리하는 설계가 필요하다.

### 5. 기능

논문이 정리한 Field Desktop의 주요 기능은 다음과 같다.

- 데스크톱 클라이언트 간 데이터베이스와 이미지 파일 동기화
- 계층형 리소스 관리와 기본 검색
- 지리정보와 기록 자료의 결합 표시
- 이미지 가져오기, 리소스 연결, 지오리퍼런싱 후 지도 레이어 활용
- 형식 자료를 계층형 목록으로 관리하고 유물과 연결
- 프로젝트 구성 편집기와 값 목록 편집기
- 리소스 관계를 기반으로 자동 생성되는 단순 매트릭스 뷰
- CSV, GeoJSON, Shapefile, 형식 카탈로그 가져오기/내보내기
- 프로젝트 백업 생성과 복구
- UI 및 데이터 입력의 다국어 지원

한국 고고학 야장 관점에서는 사진, 위치, 층위 관계, 형식 분류, 백업, 내보내기를 하나의 기록 흐름 안에 묶는 점이 특히 중요하다.

### 6. Field Hub와 iDAI.field Web

Field Hub는 기관이 연구자들에게 제공할 수 있는 중앙 동기화 대상으로 설명된다. Elixir로 작성된 작은 서버 애플리케이션이며, Docker 이미지로 배포할 수 있다. 특히 고해상도 이미지가 제대로 서버에 올라왔는지 확인하는 모니터링 기능이 중요하게 언급된다.

iDAI.field Web은 Field Desktop으로 만든 프로젝트를 공개하기 위한 플랫폼이다. 지도에서 프로젝트를 확인하고 여러 프로젝트를 가로질러 검색하는 기능을 제공한다. 논문은 이 부분이 아직 DAI 인프라 의존성을 일부 가지고 있어, 더 넓게 쓰이려면 추가 분리가 필요하다고 설명한다.

### 7. 개발 방법론

개발은 2주 단위 스프린트와 현장 프로젝트 피드백을 기반으로 진행되었다. 초기에 실제 발굴 프로젝트에서 사전 릴리스 버전을 사용하게 하고, 그 피드백을 다음 반복 개발에 반영했다는 점이 강조된다. 논문은 이것이 사용자의 실제 필요와 동떨어진 소프트웨어가 되는 것을 막는 방법이었다고 본다.

또한 코드 복잡도를 관리하기 위해 Clean Code 원칙, 리팩터링, 코드 리뷰, 단위 테스트와 E2E 테스트를 사용했다고 설명한다. 현장 기록 소프트웨어는 데이터 손실이나 동기화 오류가 치명적이므로, 이 테스트 문화는 단순한 개발 취향이 아니라 운영 안정성의 일부로 볼 수 있다.

### 8. 오픈소스화

버전 2 이후 iDAI.field는 오픈소스 라이브러리에 기반하고, 소프트웨어 자체는 Apache License 2.0으로 공개되었다. 다만 초기에는 DAI와 가까운 프로젝트 중심으로 쓰였고, 프로젝트 구성을 바꾸려면 JSON 파일을 개발자에게 전달해 앱 릴리스에 포함해야 했다.

버전 3.0부터는 프로젝트 구성이 데이터베이스 안에 저장되고, 데스크톱 앱 안의 구성 편집기로 수정할 수 있게 되었다. 이는 외부 연구자들이 DAI 개발팀을 통하지 않고도 자신들의 기록 체계를 구성할 수 있게 하는 중요한 변화다.

## 한국 고고학 야장 설계에 주는 단서

### 표준 항목과 현장 항목을 분리하기

한국형 야장을 만든다면 `core`에는 조사명, 조사기관, 조사구역, 유구/유물/시료/사진/도면의 기본 식별자, 좌표, 관계, 작성자, 작성일 같은 최소 공통 항목을 둬야 한다. 반면 지역별 시대 구분, 현장별 세부 분류, 기관별 관리 코드는 `library` 또는 프로젝트별 확장으로 다루는 편이 좋다.

### 오프라인 우선 설계

발굴 현장은 인터넷 품질이 일정하지 않다. Field가 PouchDB/CouchDB 복제를 핵심으로 둔 이유처럼, 한국형 야장도 현장에서는 오프라인으로 완전히 작동하고 사무실이나 숙소에서 지연 동기화할 수 있어야 한다.

### 사진과 공간정보를 기록의 주변부로 두지 않기

논문은 사진 관리와 지오데이터 통합을 핵심 기능으로 본다. 한국 현장에서도 사진, 도면, 좌표, 유구/유물 기록이 서로 다른 폴더와 엑셀 파일에 흩어지면 나중에 검증과 보고서 작성이 어려워진다. 기록 항목과 이미지, 지도 레이어를 처음부터 연결해야 한다.

### 값 목록과 용어집을 관리하기

한국어 현장 기록에서는 같은 개념이 기관마다 다른 용어로 쓰일 수 있다. Field의 값 목록 편집기와 library 개념은 발굴유형, 토층색, 토성, 유물 재질, 보존상태 같은 통제어휘를 관리하는 데 유용한 방향이다.

### 보고서와 공개까지 거꾸로 설계하기

논문은 1차 자료 공개와 출판 흐름을 요구사항에 포함한다. 야장 설계도 입력 화면만 볼 것이 아니라, 보고서 표, 사진대장, 유물대장, GIS 내보내기, 장기 보존, 공개 데이터셋까지 이어지는 경로를 먼저 정해야 한다.

### 현장 피드백을 개발 사이클 안에 넣기

DAI는 실제 현장 프로젝트의 피드백을 개발 반복에 반영했다. 한국형 야장도 개발자가 양식을 상상해서 만드는 방식보다, 실제 조사원이 테스트 현장에서 입력해보고 막히는 지점을 짧은 주기로 반영하는 방식이 적합하다.

## 이 저장소에서 바로 참고할 부분

- 한국어 UI와 도움말은 Field의 다국어 전략을 실제로 확인하는 기준점이 된다.
- `core/config/Library`의 범주와 값 목록은 한국형 `library`를 설계할 때 비교 대상으로 쓸 수 있다.
- `desktop/src/manual/manual.ko.md`는 사용자 교육 문서의 첫 형태로 쓸 수 있다.
- `docs/wiki/ko`의 Field Hub 문서는 기관 단위 동기화 서버를 검토할 때 같이 읽으면 좋다.
