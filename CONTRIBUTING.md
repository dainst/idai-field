# Field / iDAI.field 

Field Desktop, Field Hub, Field Web and Field Publication are Open Source tools for the documentation of archaeological (field) research. These components are developed as a cooperation between the German Archaeological Institute ([DAI](https://www.dainst.org)) and the Head Office of the GBV Common Library Network ([GBV](https://en.gbv.de/)). The main and original version of Field / iDAI.field and its components is maintained by the [*Referat für Softwaretechnik/Archäoinformatik* of the German Archaeological Institute](https://www.dainst.org/dai-standorte/zentrale/wer-wir-sind/organisation/zwd/archaeoinformatik). 

We welcome contributions to any of the Field components! Please make sure to check out the open issues in the repository and take a look at the discussions. If you are unsure whether your idea fits in with our vision for the future, do not hesitate to open a discussion and ask. 

## How do I contribute?

There are many ways to contribute:

- Submit a bug in [GitHub Issues](https://github.com/dainst/idai-field/issues).
- Start a discussion in [GitHub Discussion](https://github.com/dainst/idai-field/discussions).
- Submit a Pull Request with proposed changes, or work on an open ticket. (Feel free to get in touch with us via Discussions first, we cannot merge all suggestions.)
- Join our [Field Assembly](https://community.nfdi4objects.net/assemblies/field) on the NFDI4Objects Community Hub.

## How do I claim an Issue?

Becoming an assignee to an issue follows this workflow:

1. Comment on a thread for an open issue that you're interested in being assigned to.
2. Wait for our response. Sometimes, there may need to be further discussions on the scope as not all issues are always self-contained and self-explanatory (yet!). 
3. Fork & work on the PR for your issue in a dedicated branch, and when you're ready, submit - we will review and discuss with you.  
4. Please note that we are not a very large team - things may take some time. 

## How can I test my changes before submitting a PR?

See the respective README.md of each project on how to run tests locally. 

## Any requirements for a Pull Request (PR)?

- Before making a PR, make sure you pushed your changes from a branch other than `master`.
- Name the new branch after the changes are pushed to the PR and link the corresponding issue. 
- Do not introduce new dependencies. (Or discuss with us first, if you must.)
- Follow the Style Guide

## Can I use AI tools when contributing?

You can use AI tools. But AI should help you be more efficient, not do all the work for you. We expect you to: 

- diligently review all generated content before submitting
- understand and approve of each line of code you submit
- be able to answer for and explain to us all of your changes and their logic

You are accountable for all AI-assisted content, be it code or text. If you cannot answer for your changes, we cannot accept your work. 

AI can often add unneeded abstractions, overcomplicate simple functionalities, select less self-explanatory variable names, and generate redundant documentation / comment string. Make sure you address related issues before submission.
